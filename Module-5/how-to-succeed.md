# How to Succeed With Larger Front-End Assignments

This assignment is not only about *what* you build, but *how* you work.
Strong projects come from deliberate planning, structured execution, and clear documentation - not from rushing into code.

Use the checklist below **before**, **during**, and **after** development.

## 1. Read the Assignment Carefully (Yes, Really)

Before writing a single line of code:

* Read the **entire assignment text from top to bottom**
* Read it **again**
* Highlight or note:

  * Mandatory requirements
  * Optional / bonus features
  * Explicit constraints (tools, frameworks, APIs, time limits)
  * What is *not* allowed

Ask yourself:

* What is being assessed?
* What would count as a *minimum pass*?
* What would distinguish a *strong* submission from an average one?

If something is unclear:

* Ask early
* Ask specifically
* Quote the exact part you are unsure about

## 2. Understand the Learning Outcomes

Every larger assignment exists to test specific skills.

Make sure you can clearly answer:

* Which **technical skills** is this meant to demonstrate?
* Which **concepts** are more important than others?
* What are you expected to explain or defend if asked?

If you cannot explain *why* a certain technique is used, you are probably not done.

## 3. Plan Before You Implement

Do not code first and hope it works out later.

Before implementation, you should have:

### Functional Overview

* What features does the application need?
* What user actions are required?
* What data is displayed, created, or modified?

### Technical Plan

* Folder / file structure
* Main components or modules
* State management approach (if relevant)
* Data flow (API → state → UI)

### Scope Control

* What is **required**
* What is **nice to have**
* What will be dropped if time runs short

A smaller, complete solution beats a half-finished ambitious one.

## 4. Set Up the Project Properly

Before building features:

* Initialize the project cleanly
* Ensure:

  * The app runs locally without errors
  * Linters / formatters (if used) are working
  * Git is initialized early
* Commit small, meaningful changes frequently

Avoid:

* One giant final commit
* Committing broken code
* Leaving unused files or commented-out experiments

## 5. Implement Incrementally

Build in small, testable steps:

* One feature at a time
* One component/module at a time
* Verify functionality continuously

If something breaks:

* Stop
* Fix it
* Then continue

Do not pile new code on top of unresolved problems.

## 6. Write Code for Humans

Your code should be understandable to:

* You in two weeks
* Another student
* An examiner

Guidelines:

* Use clear, descriptive names
* Avoid unnecessary cleverness
* Keep functions/components focused
* Remove dead code

If a solution is complex:

* Add a short comment explaining *why*, not *what*

## 7. Handle Errors and Edge Cases

Even simple applications should show care:

* Handle loading states
* Handle errors from APIs or user input
* Avoid crashes on missing or unexpected data

Perfection is not required - but awareness is.

## 8. Test What You Build

Before submission:

* Click through every feature
* Try invalid input
* Reload the page
* Test on different screen sizes
* Ensure nothing obvious is broken

If the assignment includes automated testing:

* Write the required unit/integration/E2E tests
* Make sure they actually run
* Do not submit failing tests

## 9. Document Your Work (README Matters)

Your README is part of the delivery.

It should include:

* Short project description
* How to run the project locally
* Technologies used
* Any known limitations or missing features
* If AI tools were used: how and for what

A good README signals professionalism and control.

## 10. Review Against the Assignment (Again)

Before submitting:

* Re-read the assignment text
* Verify every requirement is addressed
* Check naming, structure, and clarity
* Remove unfinished or broken features

Ask yourself:

> “If I were grading this, would I clearly see that all requirements are met?”

## 11. Final Mindset

Strong projects are not rushed.
They are:

* Planned
* Controlled
* Thought through
* Finished

Coding fast is not the goal.
Delivering a clear, complete, explainable solution is.
